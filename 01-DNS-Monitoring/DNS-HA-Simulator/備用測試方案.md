# 備用測試方案

如果 port forward 到 53 端口有問題，可以使用以下備用方案：

## 方案 1: 直接使用 5300 端口測試

```bash
# 直接查詢 Docker 映射的 5300 端口
dig @127.0.0.1 -p 5300 app.example.com

# 或使用 nslookup
nslookup app.example.com 127.0.0.1 -port=5300
```

## 方案 2: 修改 macOS DNS 設定使用非標準端口

雖然 macOS 系統 DNS 設定不支援指定端口，但可以：

1. **使用 dscacheutil 測試**：
   ```bash
   # 這不會工作，因為系統 DNS 不支援自定義端口
   ```

2. **使用 hosts 文件（臨時測試）**：
   ```bash
   # 編輯 /etc/hosts，添加：
   # 172.20.0.10 app.example.com
   # 但這會繞過 DNS 查詢，無法測試 DNS 切換邏輯
   ```

## 方案 3: 使用 Docker 容器測試

在 Docker 容器內測試，完全繞過 macOS DNS：

```bash
# 進入 client 容器
docker exec -it dns-client sh

# 在容器內測試
dig @172.20.0.101 app.example.com
curl http://app.example.com
```

## 方案 4: 修改 Docker 端口映射（需要 root）

如果必須使用 53 端口，可以修改 Docker 配置：

```bash
# 修改 docker-compose.yml，將 dns-secondary 的端口改為：
# ports:
#   - "53:53/udp"
#   - "53:53/tcp"

# 然後重啟容器（需要 root 權限）
sudo docker-compose up -d dns-secondary
```

**注意**：這需要 Docker 有權限綁定 53 端口，可能需要特殊配置。

## 推薦方案

**對於測試 DNS HA 功能**，建議使用 **方案 3（Docker 容器測試）**，因為：
- 不需要修改 macOS 系統設定
- 不需要處理端口衝突
- 可以完整測試 DNS 切換邏輯
- 容器內的測試結果與實際環境更接近

**對於在 macOS 主機上測試**，如果 port forward 仍有問題，建議：
1. 先使用方案 1 直接測試 5300 端口，確認 DNS 服務正常
2. 如果必須使用 53 端口，考慮使用方案 4（需要 Docker 特殊配置）
