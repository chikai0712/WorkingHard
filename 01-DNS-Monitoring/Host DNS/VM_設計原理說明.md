# 自建 DNS 環境設計原理（中型 20k QPS）

## 一、設計原則

### 1. 高可用（HA）設計
- **目標**：任一實體主機故障時，服務仍可正常運作
- **策略**：4 台 DNS VM 分散在 3 台實體主機上
  - 實體主機 #1：2 台 DNS VM + 1 台監控 VM
  - 實體主機 #2：2 台 DNS VM + 1 台 Bastion VM
  - 實體主機 #3：備用（可跑 1 台 DNS VM 或關機）
- **結果**：任何 1 台實體主機故障，仍有 2-3 台 DNS VM 運作（足夠 20k QPS）

### 2. 資源分配邏輯

#### 實體主機規格（每台）
- **CPU**：雙路 16 核心 = 32 實體核心（64 vCPU）
- **RAM**：128GB
- **SSD**：2×960GB（RAID1）= 960GB 可用（放 VM 系統碟）
- **HDD**：4×4TB（RAID10）= 8TB 可用（放日誌/DNSTAP）

#### VM 資源分配
- **DNS VM**：8 vCPU / 16GB（每台可處理 8k-12k QPS）
- **監控 VM**：4 vCPU / 16GB（Prometheus + Loki）
- **Bastion VM**：2 vCPU / 4GB（管理用）

### 3. 容量估算

#### 正常運作
- 4 台 DNS VM × 10k QPS = **40k QPS 能力**
- 足夠應付 **20k QPS 峰值** + 2x 安全冗餘

#### 故障場景
- 1 台實體主機故障 → 剩 2-3 台 DNS VM → **20k-30k QPS 能力**
- 仍足夠應付 20k QPS 峰值

## 二、硬碟使用量估算

### DNS VM（每台）
- **系統碟**：100GB（OS + dnsdist + Unbound）
- **日誌碟**：200GB（DNSTAP 原始封包，保留 7 天）
- **總計**：300GB

### 監控 VM（1 台）
- **系統碟**：300GB（Prometheus + Loki + 系統）
- **日誌碟**：1000GB（Prometheus metrics 90 天 + Loki 日誌 90 天）
- **總計**：1300GB

### 實體主機硬碟配置
- **SSD（RAID1）**：960GB 可用 → 放所有 VM 系統碟
- **HDD（RAID10）**：8TB 可用 → 放所有日誌/DNSTAP

**使用率估算**：
- 實體主機 #1：SSD 500GB / 960GB = 52%，HDD 1400GB / 8000GB = 18%
- 實體主機 #2：SSD 200GB / 960GB = 21%，HDD 450GB / 8000GB = 6%
- 實體主機 #3：SSD 100GB / 960GB = 10%，HDD 200GB / 8000GB = 3%

## 三、成本結構

### 一次性成本（CAPEX）
- 3 台實體主機硬體
- 虛擬化授權（如 VMware）
- 機櫃與電力（首年）

### 持續成本（OPEX）
- 機房空間與電力（月費）
- 維護合約（年費）
- 折舊攤提（硬體 5 年、授權 3 年）

**月成本 = 硬體折舊 + 授權折舊 + 機房費用**

## 四、與雲端方案對比

| 項目 | 自建（3台實體主機） | AWS（4台 m6i.2xlarge） | GCP（4台 n2-standard-8） |
|------|-------------------|----------------------|------------------------|
| 初期成本 | 高（需買硬體） | 低（按需付費） | 低（按需付費） |
| 月成本 | 約 $1,000-2,000 | 約 $1,340 | 約 $1,410 |
| 彈性 | 低（需擴充硬體） | 高（隨時調整） | 高（隨時調整） |
| 控制權 | 高（完全控制） | 中（受限於雲端） | 中（受限於雲端） |

## 五、建議

### 適合自建的情況
- 長期穩定需求（3-5 年）
- 需要完全控制硬體與資料
- 已有機房與維運團隊
- 預算允許一次性投資

### 適合雲端的情況
- 需求可能快速變化
- 不想管理硬體
- 需要快速擴充/縮減
- 初期預算有限
