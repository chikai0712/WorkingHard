## DNS 方案分析整理

本文件彙整先前討論的兩部分內容：
- 多組 Name Server 的故障行為與對玩家體感的影響
- 各常見權威 DNS 方案（AWS、GCP、NS1、Akamai、自建）的費用與優缺點

---

### 一、Name Server 故障行為與玩家體感
- 解析流程：玩家端 → 遞迴 DNS → 根/TLD → 註冊商回傳多組 NS → 權威 DNS 回應 → 遞迴快取 → 玩家連線。
- 多組 NS 的查詢方式：遞迴 DNS 會在 TLD 回傳的 NS 清單中選擇可用的 NS；若其中一台逾時會改問下一台，並暫時標記故障 NS。
- 主流遞迴伺服器（如 BIND、Unbound、Google Public DNS）在遇到失敗時會有 retry/backoff 機制：以 BIND 為例，對同一 NS 連線失敗後會立刻改詢其他 NS，並將失敗的 NS 暫時「降權」；對同一問題通常最多重試 2–3 次，失敗後會在 15 分鐘（可調）內視為「lame server」而不再詢問，直到 TTL 或觀察期結束。
- RFC 1034/1035 也建議遞迴端對 NXDOMAIN 或 timeout 進行多次重試，並且在多個 NS 間隨機化查詢，避免單點。
- Downtime 情況：
  - 只要至少一組 NS 可用，實際 downtime 約等於 0，最多首次查詢有短暫延遲。
  - 真正 downtime 發生在所有 NS 都不可用且原快取 TTL 到期後，才會出現 NXDOMAIN/逾時。
- 玩家感受：若仍有可用 NS，通常無感；所有 NS 都掛時會看到「DNS 查詢逾時 / 網站無法載入 / DNS_PROBE_FINISHED_NXDOMAIN」等錯誤。
- 建議：多組 NS 分散供應商/機房/ASN，TTL 不要過短，監控 NS 健康，必要時用 Anycast 或多雲權威 DNS。

### 二、權威 DNS 方案費用與優缺點（USD）

| 方案 | 費用（常見公開價） | 優點 | 缺點 |
| --- | --- | --- | --- |
| AWS Route 53 | ~$0.50/zone/月；~$0.40/百萬查詢（前 10 億）；健康檢查 ~$0.50/檢查/月 | 全球 Anycast、穩定；健康檢查與路由策略齊全；與 AWS 服務整合 | 高級防護需另搭 AWS 服務；超大流量帳單可觀 |
| Google Cloud DNS | ~$0.20/zone/月；~$0.40/百萬查詢 | 全球 Anycast、延遲低；API/介面簡單；價格透明 | 流量導流策略較基本；高級防護需 Cloud Armor 等 |
| NS1 (IBM NS1 Connect) | 企業合約制，依查詢量與功能打包 | 強大即時流量導控與多雲導流；API 自動化成熟 | 公開價不透明，小量成本高，需學習高階策略 |
| Akamai Edge DNS | 企業合約制，通常固定月費＋查詢階梯 | 邊緣覆蓋廣、抗 DDoS 強；企業 SLA；與 Akamai 產品整合 | 價格談判式，小量性價比低；自助度較低 |
| 自建權威 DNS | 主機/頻寬/DDoS 防護/人力：小規模約 $50–$200+/月，視機房與防護而定 | 完全客製與控管；無查詢計量費 | 維運與防護責任自負；需多區、多雲部署與專人；遇大流量攻擊風險與成本高 |

**適用情境建議**
- 入門/彈性計價：AWS 或 GCP，快速啟用、多區與低門檻。
- 進階流量導控、多雲分配：NS1。
- 強邊緣與企業防護/SLA：Akamai Edge DNS（若已使用其 CDN/WAF 產品）。
- 完全自控且有網工/資安團隊：自建，但務必加上 DDoS 防護與多點部署。

### 三、測試與驗證方案（建議流程）
- **權威 NS 可用性**：對各 NS 做 `dig +trace` 與 TCP/UDP 53 連線檢查，測試跨區/跨 ISP。驗證 NSSEC（如有啟用）是否一致。
- **lame server 檢測**：刻意關閉其中一台 NS，觀察遞迴端（BIND/Unbound/公共 DNS）是否快速切換並標記 lame；確認其他 NS 正常回答。
- **TTL 與快取行為**：設定短 TTL（如 60s），多地反覆查詢，看更新傳播時間與遞迴快取行為；同時記錄失敗時的回退延遲。
- **流量導流與健康檢查**：對具導流策略（加權/延遲/地理）的服務，模擬一個後端故障，觀察權威回覆是否切換到健康節點；驗證健康檢查頻率與切換時間。
- **高併發與解析延遲**：以 dnsperf/k6 模擬高 QPS，量測 P50/P95/P99 延遲；同時監控權威與遞迴端資源使用率與丟包率。
- **跨供應商一致性**：若多供應商混合（例如 AWS + GCP + NS1），比對各家回覆是否一致（IP/TTL/記錄集）；可用 `dnsdiag` 或自寫腳本輪詢。
- **故障演練**：模擬單台 NS、單供應商、單區域故障，驗證：
  - 遞迴端的重試次數與降權時間（觀察 query log）
  - 使用者端是否只出現偶發延遲而非全面宕機
- **監控與告警驗證**：確認權威 NS 的監控指標（可用率、延遲、SERVFAIL、NXDOMAIN 比例）與告警門檻；觸發一次測試告警確保通知通暢。
