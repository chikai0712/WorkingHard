# 災難復原 (DR) 執行劇本

本劇本旨在確保在災難發生時，技術團隊能按照標準作業程序 (SOP) 迅速恢復服務，最小化業務中斷時間,。

## 第一階段：UM 啟動前任務 (PRE UM TASK)
**目標：完成初期通報與任務分配。**

1.  **內部通報：** 向 所有產品負責人 (PO) 報告災情並啟動復原流程,,。
2.  **通知客戶：** 由 WL 團隊負責通知 NS 客戶，並發布維護 (UM) 訊息,,。
3.  **團隊就位：** 
    *   **IDC：** 負責雲端機器、網路與 其他相關 腳本執行,,。
    *   **IDD：** 負責 Jenkins/GitLab 環境與代碼還原,,。
    *   **APS/LCG/WL：** 負責應用程式服務與關鍵路徑驗證，注單相關驗證DR 資訊提供,,。

---

## 第二階段：系統恢復執行 (UM START)
**目標：重建基礎設施、還原資料庫並釋放應用程式。**

### 1. 基礎架構與網路設置
*   **掛載維護頁面：** 透過 Cloudflare 將流量導向 UM 頁面（例如 `um.dryy0.com`）,,,。
*   **開啟 DR 伺服器：** 根據「2023 DR Server List」開啟 AWS EC2 實例,,。 
    *   *注意：* 若自動化開啟失敗（容量不足），需改為**分批手動開啟**。
*   **網路切換：** 執行內部 DNS 腳本，並將公共 DNS 記錄從 Cloudflare 切換至 CDNetworks,,。

### 2. 代碼與部署環境恢復
*   **還原 Git 伺服器：** 從 AWS AMI 鏡像恢復 `sp-git-gitlab` 與 `wl-git-gitlab`,,。
*   **配置 Git Lab Runner：** 修改 `config.toml` 文件，將生產環境 IP 指向 DR Git 伺服器，並重啟服務,,,。

### 3. 資料庫同步與還原 (DB Restore)
*   **S3 資料同步：** 使用 Jenkins Job 將資料庫備份檔案從 S3 同步至 DB 磁碟,,,。
*   **核心資料庫還原：** 依序還原 SportsBook Main DB、sp-operation、VirtualSports 等,,,。
*   **YY 站點還原：** 還原 YY1-5 的 NS 與 OS 資料庫,,,。
*   **權限與狀態檢查：** 
    *   確認資料庫擁有者 (Owner) 已變更為 **sa**,,,。
    *   檢查最新注單時間 (Latest bet order time) 以確保資料完整性,,,。

### 4. 應用程式釋放 (Application Release)
*   **釋放 OS 應用程式：** 包含 Triton, Nike, Elpis, FrontEnd (Castor/Dragon/Kratos) 等,,,。
*   **釋放 NS 應用程式：** 包含 Mirana, YogApi, Angelia 等,,,。
*   **Pluto 狀態檢查：** 確認 Pluto 同步狀態，必要時手動重啟。應在 **15:30 (GMT+8)** 歸檔作業完成後執行，以避免同步緩慢,,。

---

## 第三階段：路徑驗證與測試 (Validation)
**目標：確認系統功能正常，可對外提供服務。**

1.  **APS 關鍵路徑測試：** 使用測試帳號 (如 `19yy_eztest`) 於 YY 站點進行登入與下注測試,,,。
2.  **LCG 關鍵路徑測試：** 驗證第三方供應商 (Naga, SexyGaming, BigGaming) 的連線與遊戲啟動,,,。
3.  **F5/SAPI 重定向：** 請求 IDC 調整第三方 F5 記錄，將流量正式導向 DR 環境,,,。

---

## 第四階段：恢復後期任務 (AFTER UM)
**目標：移除限制並鞏固環境穩定性。**

1.  **移除維護頁面：** 測試完成後，移除 Cloudflare 的維護頁面重定向規則,,,。
2.  **環境清理：** 刪除 DR VM 上的所有舊 Log 檔案，避免磁碟空間不足,,,。
3.  **建立完整備份：** 在環境穩定運作後，立即執行一次 Full Backup,,,。
4.  **高可用性 (HA) 設置：** 完成 YY 的 HA 機制配置，確保 DR 環境具備冗餘能力,,,。

---

## 重要回顧建議與注意事項 (Retro Insights)

*   **伺服器規格：** **Nike 服務** 至少需要 `t3.xlarge` 規格才能穩定運行,。
*   **管理機性能：** 建議將管理伺服器 (mgmt) 記憶體提升至 **24 GB**，並在演練前強制登出閒置用戶，以防止還原腳本崩潰。
*   **自動化改進：** 建議將「資料庫還原」、「修正用戶」與「變更擁有者」整合為單一腳本，減少人工操作失誤,。
*   **日誌管理：** 由於磁碟空間限制，需定期執行刪除 Log 檔案的任務,。
