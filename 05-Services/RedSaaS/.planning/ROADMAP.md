# RedSaaS — 博弈產業資安自動化平台 ROADMAP

## 產品定位

針對博弈產業的 **攻擊面管理 + AI 行為防禦 + 合規報告自動化** 三合一平台。
先內部使用（對自有測試環境），再對外銷售給博弈業者與資安公司。

**核心差異化**：
- 內建博弈業務邏輯漏洞 Template 庫（Nuclei + secureCodeBox BAS）
- 符合 MGA/PAGCOR 合規格式的中文報告自動生成
- **AI 行為基線防禦**：每日吞吐交易日誌，建立 UI 操作基準，偵測「跳步驟」異常行為
- **雙層 AI 架構**：前線哨兵（1B/8B，<30ms）+ 後台推理大腦（地端 70B，根因分析）

## 關鍵設計原則

1. **商業邏輯基準優先**：不只掃技術漏洞，要建立「正常 UI 操作流程的時序基準」
2. **閉環自動化**：BAS 攻擊手法在 24 小時內自動轉化為防禦規則更新
3. **地端部署**：交易資料不出企業內網（PII + 營業秘密合規）
4. **雙層分工**：輕量模型守即時，大模型做審計 — 速度與深度兼顧

## Phase 1｜內部工具化（0-6 個月）

**目標**：對內部測試環境可以自動跑掃描、出中文報告草稿，省掉 60% 手工整理時間。

### 1-01 本地測試環境建立
- [x] 部署 OWASP crAPI（API 漏洞靶場）
- [x] 部署 DVWA（通用網頁漏洞靶場）
- [ ] 部署一個開源博弈後台（含故意未修補的漏洞）
- [x] 安裝 Nuclei + secureCodeBox 並驗證可正常執行
- [x] 安裝 DefectDojo 作為漏洞管理後台
- **驗證方式**：對 crAPI 跑一次 Nuclei 掃描，結果可在 DefectDojo 看到 ✅ 完成（Test ID: 1，10 findings）

### 1-02 博弈漏洞 Template 庫初版
- [x] 金流 API 越權（IDOR/BOLA）— 玩家餘額、交易紀錄
- [x] 代理商後台 Rate Limiting 缺失
- [x] 存提款 API 金額邊界值驗證
- [x] 玩家 API 認證 Token 洩漏偵測
- [x] 後台管理介面預設路徑暴露
- **驗證方式**：在本地靶場跑，確認 True Positive / False Positive 比率 ← 下一步（需真實博弈平台路徑）

### 1-03 報告自動生成器（MVP）
- [x] DefectDojo API 串接，抓取結構化漏洞資料
- [x] 用 OpenAI API 或本地 Ollama 將漏洞資料轉成中文段落
- [x] 輸出 Word/PDF 報告草稿（符合現有報告格式）
- **驗證方式**：一份報告從掃描完成到草稿輸出，時間在 15 分鐘內 ✅ 完成（約 3 分鐘）

### 1-04 基礎掃描排程
- [ ] Docker Compose 管理所有工具容器
- [ ] Celery + Redis 處理掃描任務隊列
- [ ] 簡單的 CLI 或最小化 Web UI：輸入目標 → 觸發掃描 → 看結果
- **驗證方式**：一個非資安背景的人能獨立操作完成一次掃描

## Phase 1.8｜AI 行為防禦層（新增）

**目標**：建立基於交易日誌與 UI 操作時序的行為基線，偵測「繞過正常業務流程」的攻擊。
這是對話中「每天聽不完的內部資訊 → 商業邏輯 UI 基準」概念的落地實作。

### 1.8-01 資料收集管線
- [ ] 定義需收集的日誌類型：API 呼叫鏈、UI 操作時序、交易紀錄
- [ ] 將結構化日誌「文字化（Textify）」：`user=123, action=deposit, amount=500, ts=...` → 自然語言描述
- [ ] 建立 Log 儲存格式（PostgreSQL 或 ClickHouse）
- **驗證方式**：每天可自動收集靶場環境的操作日誌，格式一致

### 1.8-02 第一層前線哨兵（輕量模型 1B/8B）
- [ ] 選定基底模型（Ollama 本地部署，如 llama3.2:1b 或 qwen2.5:3b）
- [ ] 用 BAS 攻擊日誌 + 正常操作日誌建立訓練樣本
- [ ] 微調：教模型分辨「人類正常操作時序」vs「程式化 API 暴力呼叫」
- [ ] 部署為 REST API，延遲目標 < 50ms
- **驗證方式**：BAS 模擬「1 秒內呼叫存款 API 100 次」時，哨兵在 30ms 內觸發告警

### 1.8-03 第二層後台推理大腦（地端 70B）
- [ ] 選定基底模型（Qwen2.5-72B 或 Llama3.3-70B，地端 GPU 部署）
- [ ] RAG 知識庫：將博弈業務規則手冊 + ROADMAP + 已知攻擊模式向量化
- [ ] LoRA 微調：用真實攻防案例教模型串聯「跨 API 的商業邏輯矛盾」
- [ ] 第一層觸發告警後，70B 自動調取前後 24 小時日誌做根因分析
- [ ] 自動產出中文事件報告（根因 + 影響範圍 + 建議處置）
- **驗證方式**：一個複合式攻擊事件（BAS 觸發），5 分鐘內產出可讀的中文根因報告

### 1.8-04 UI 行為基線建立
- [ ] 定義各關鍵流程的「正常操作時序」：
  - 登入 → 瀏覽 → 存款：最短間隔 X 秒，步驟不可跳過
  - 代理商後台操作：正常 API 呼叫頻率基線
  - 提款申請：必須經過哪些前置步驟
- [ ] 把基線規則寫入 RAG 知識庫
- [ ] 測試：BAS 模擬「繞過 UI 直接呼叫後端結帳 API」能否被偵測
- **驗證方式**：直接呼叫後端結帳 API（不走前端 UI 流程）→ 第一層哨兵在 50ms 內觸發

**目標**：讓博弈業者 IT 主管可以自助操作，建立月費制訂閱收入。
差異化賣點：同時提供「紅隊掃描」與「AI 行為防禦」兩條服務線，市場上目前無競品。

### 3-01 多租戶架構
- [ ] 從 Docker Compose 遷移到 K8s（GKE 或 EKS）
- [ ] 租戶隔離：每個客戶的掃描在獨立容器內執行
- [ ] 動態 IP 輪替（Egress Proxy）
- [ ] 地端 70B 部署指南（客戶自備 GPU 的安裝文件）

### 3-02 Self-service Web UI
- [ ] 客戶自助操作介面（React + FastAPI）
- [ ] 多語言支援（繁中、簡中、英文）
- [ ] AI 防禦看板：每日行為基線 + 異常事件時序圖

### 3-03 計費與帳號管理
- [ ] 訂閱方案設計（紅隊掃描版 / 全套攻防版）
