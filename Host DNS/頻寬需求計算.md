# DNS 服務頻寬需求計算（20k QPS）

## 一、瞬間頻寬計算

### 基本計算公式
```
頻寬 (Mbps) = QPS × 封包大小 (bytes) × 8 (bits/byte) / 1,000,000
```

### 20k QPS 頻寬需求

#### 入站頻寬（查詢）
- **查詢封包**：50 bytes
- **頻寬**：20,000 × 50 × 8 / 1,000,000 = **8 Mbps**

#### 出站頻寬（回應）
- **回應封包**：80 bytes
- **頻寬**：20,000 × 80 × 8 / 1,000,000 = **12.8 Mbps**

#### 總頻寬（雙向）
- **總計**：8 + 12.8 = **20.8 Mbps**

---

## 二、實際頻寬需求建議

### 考慮因素
1. **峰值突發**：實際流量可能瞬間超過平均值 1.5-2 倍
2. **安全邊際**：建議預留 20-30% 緩衝
3. **監控流量**：Prometheus + Loki 約 0.5-1 Mbps
4. **管理流量**：SSH + API 約 0.1 Mbps（可忽略）

### 建議頻寬配置

| 項目 | 基本需求 | 建議配置 | 說明 |
|------|---------|---------|------|
| **出站頻寬** | 12.8 Mbps | **20-25 Mbps** | 主要頻寬需求（回應流量） |
| **入站頻寬** | 8 Mbps | **15-20 Mbps** | 查詢流量（通常免費） |
| **總頻寬** | 20.8 Mbps | **35-45 Mbps** | 含安全邊際 |

---

## 四、雲端頻寬配置

### AWS EC2
- **標準頻寬**：依實例類型而定
  - `m6i.2xlarge`：最高 **12.5 Gbps**（足夠）
  - `m6i.xlarge`：最高 **12.5 Gbps**（足夠）
- **網路效能**：可處理遠超過 20k QPS 的需求

### GCP Compute Engine
- **標準頻寬**：依機器類型而定
  - `n2-standard-8`：最高 **16 Gbps**（足夠）
  - `n2-standard-4`：最高 **16 Gbps**（足夠）
- **網路效能**：可處理遠超過 20k QPS 的需求

---

## 五、結論

### 20k QPS 環境頻寬需求
- **瞬間出站頻寬**：**12.8 Mbps**
- **瞬間入站頻寬**：**8 Mbps**
- **總頻寬**：**20.8 Mbps**
- **建議配置**：**35-45 Mbps**（含安全邊際）

### 雲端實例頻寬能力
- **AWS / GCP 標準實例**：最高 12.5-16 Gbps
- **結論**：**完全足夠**，無需額外頻寬費用

---

## 六、注意事項

1. **頻寬 vs 流量**：
   - **頻寬**：瞬間傳輸速度（Mbps）
   - **流量**：累積傳輸量（GB/月）
   - 20k QPS 需要 12.8 Mbps 頻寬，但每月產生 4,147 GB 流量

2. **雲端頻寬計費**：
   - AWS/GCP 通常**不單獨計費頻寬**
   - 只計費**出站流量**（GB）
   - 頻寬是實例的內建能力

3. **瓶頸分析**：
   - 20k QPS 的瓶頸通常是 **CPU** 而非頻寬
   - 標準雲端實例的頻寬遠超過需求
