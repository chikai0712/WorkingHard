# 純權威DNS架構說明

## 架構概述

本架構為**純權威DNS架構**，只託管自己的域名，不進行遞迴查詢。

### 核心特點

1. **只託管自己的域名**：權威DNS只回應自己託管的域名查詢
2. **不進行遞迴查詢**：不會查詢其他域名，因此沒有遞迴查詢的流量費用
3. **只有權威DNS回應流量**：當玩家查詢您的域名時，才會產生回應流量

## 架構流程

```
Player (玩家)
    ↓ DNS查詢請求（查詢您的域名，如 example.com）
Recursive DNS Server (遞迴DNS服務器，如 8.8.8.8)
    ↓ 查詢權威記錄
自建權威DNS - BIND DNS Master
    ↓ 返回權威記錄（A記錄、AAAA記錄等）
Player (玩家)
```

### 流量產生機制

1. **查詢流程**：
   - 玩家查詢您的域名（如 `example.com`）
   - 玩家的遞迴DNS服務器（如 Google 8.8.8.8）查詢您的權威DNS
   - 您的權威DNS返回自己託管的域名記錄
   - 遞迴DNS服務器將結果返回給玩家

2. **流量說明**：
   - **查詢流量（入站）**：遞迴DNS服務器查詢您的權威DNS（通常免費）
   - **回應流量（出站）**：權威DNS返回記錄給遞迴DNS服務器（產生費用）
   - **不會產生遞迴查詢流量**：因為不查詢其他域名

3. **流量計算**：
   - 權威DNS回應封包：約 80-100 bytes
   - 20k QPS = 4,147 GB/月的出站流量
   - 相比遞迴解析器架構，流量費用不變但架構更簡單

## 技術架構

### 權威DNS服務器

**技術棧**：
- **BIND DNS**：業界標準的權威DNS服務器
- **VM規格**：n2-standard-4 (4vCPU/16GB) 或更高
- **高可用選項**：2台Active-Active架構，跨區域部署

**功能**：
- 從域名註冊商（Godaddy）同步DNS記錄
- 管理DNS記錄的權威來源
- 對外提供權威DNS查詢服務
- 只回應自己託管的域名查詢

### 與遞迴解析器的區別

| 特性 | 權威DNS | 遞迴解析器 |
|------|---------|-----------|
| **角色** | 只回應自己託管的域名 | 可以查詢任何域名 |
| **流量** | 只有權威DNS回應流量 | 權威DNS回應 + 遞迴查詢流量 |
| **VM需求** | 中型VM（n2-standard-4） | 大型VM（n2-standard-8） |
| **技術棧** | BIND DNS | dnsdist + Unbound |
| **成本** | 約 $200-400/月 | 約 $1,300-1,900/月 |
| **適用場景** | 只託管自己的域名 | 需要查詢其他域名 |

## 優勢與劣勢

### 優勢

1. **成本低**：
   - 不需要遞迴解析器（dnsdist + Unbound）
   - VM成本降低約 $400/月
   - 只有權威DNS回應流量，沒有遞迴查詢流量

2. **架構簡單**：
   - 單一角色（權威DNS服務器）
   - 職責清晰：只託管自己的域名
   - 維護簡單：不需要複雜的遞迴解析配置

3. **完全控制**：
   - 完全控制DNS記錄管理
   - 快速恢復（1-2小時）
   - 不依賴第三方服務

4. **成本優化**：
   - 可以享受雲端折扣（30%）
   - 高QPS情況下，比第三方服務便宜95%以上

### 劣勢

1. **維護成本**：
   - 需要專業運維團隊維護基礎設施
   - 需要監控和告警系統

2. **初期投入**：
   - 需要較高的初期投入和設置時間
   - 需要DNS和基礎設施管理專業知識

3. **功能限制**：
   - 只能回應自己託管的域名
   - 不能查詢其他域名（如果需要，需要額外的遞迴解析器）

## 適用場景

### 適合使用純權威DNS架構的情況

1. **只託管自己的域名**：
   - 不需要查詢其他域名
   - 只需要回應自己域名的查詢

2. **高QPS環境**：
   - QPS > 10k
   - 需要控制成本

3. **需要完全控制**：
   - 需要快速恢復能力（1-2小時）
   - 需要自定義DNS記錄管理

4. **成本敏感**：
   - 希望降低DNS服務成本
   - 可以接受自建維護成本

### 不適合使用純權威DNS架構的情況

1. **需要查詢其他域名**：
   - 如果需要查詢其他域名，需要額外的遞迴解析器
   - 這種情況下，遞迴解析器架構更適合

2. **低QPS環境**：
   - QPS < 5k
   - 第三方DNS服務可能更經濟

3. **無運維團隊**：
   - 沒有專業運維團隊
   - 無法維護基礎設施

## 部署建議

### 基本部署（單台）

1. **VM規格**：n2-standard-4 (4vCPU/16GB)
2. **作業系統**：Ubuntu 22.04 LTS 或 CentOS 8
3. **DNS軟體**：BIND 9
4. **監控**：Prometheus + Grafana
5. **備份**：定期備份DNS記錄

### 高可用部署（2台Active-Active）

1. **VM規格**：n2-standard-4 (4vCPU/16GB) × 2
2. **部署位置**：跨區域部署（如 asia-northeast1 和 asia-northeast2）
3. **負載均衡**：GCP Global TCP Load Balancer
4. **DNS同步**：使用 BIND 的 Master-Slave 或 Multi-Master 架構
5. **監控**：監控兩台VM的健康狀態

### 安全建議

1. **防火牆**：只允許 53/UDP 和 53/TCP 從互聯網訪問
2. **DNSSEC**：啟用 DNSSEC 簽名
3. **限流**：配置查詢限流，防止DDoS攻擊
4. **監控**：監控異常查詢和攻擊行為
5. **備份**：定期備份DNS記錄和配置
