# 域名檢測狀態分類系統

## 1. 狀態分類定義

### 🟢 CLEAN - 正常連通
**定義**：域名可正常解析且 HTTP 連線成功
**HTTP 狀態碼**：
- `2xx`：成功（200, 201, 202 等）
- `3xx`：重定向（301, 302, 307, 308 等）
- `403`：Forbidden（雖然被拒絕，但服務器有回應，DNS 正常）

**輸出格式**：`[CLEAN] ✅ 正常連通 (HTTP {code})`

**含義**：
- DNS 解析正常
- TCP 連線建立成功
- HTTP 服務有回應
- 不代表網站內容正常，只代表「連得上」

---

### 🔴 BLOCKED - DNS 污染/劫持
**定義**：DNS 解析到已知的污染 IP 或內網 IP
**判斷條件**：
- 解析到 `36.86.63.185`（已知污染 IP）
- 解析到 `10.*.*.*`（內網 IP，不應該是公網域名的解析結果）
- 解析到其他已知的 ISP 劫持 IP

**輸出格式**：`[BLOCKED] 🚨 DNS 污染`

**含義**：
- ISP 層級的 DNS 劫持
- 域名被主動封鎖
- 需要使用 VPN 或更換 DNS 才能訪問

**建議動作**：
- 通知營運團隊
- 考慮更換域名或使用 CDN
- 記錄封鎖的 ISP 和地區

---

### ⚠️ TIMEOUT - 連線超時/無回應
**定義**：DNS 無法解析或 HTTP 連線超時
**判斷條件**：
- DNS 查詢無結果（`resolvedAddress` 為空）
- HTTP 狀態碼為 `0`、`null`、`None`
- 連線超時無回應

**輸出格式**：`[TIMEOUT] ⚠️ 阻斷/無回應`

**可能原因**：
1. **DNS 無法解析**：
   - 域名未配置 DNS 記錄
   - DNS 服務器故障
   - 域名已過期
   
2. **網絡層阻斷**：
   - 防火牆封鎖
   - ISP 封鎖（但不是 DNS 污染）
   - 目標服務器關閉

3. **區域性問題**：
   - 特定 ISP 無法訪問
   - 地理位置限制
   - 臨時網絡故障

**分析方法**：
- 如果 3 個節點都 TIMEOUT → 可能是域名本身問題
- 如果部分節點 TIMEOUT → 可能是區域性封鎖或 ISP 問題

---

### ⚠️ WARNING - 異常狀態
**定義**：服務器有回應但狀態異常
**HTTP 狀態碼**：
- `4xx`（除了 403）：客戶端錯誤（400, 404, 405 等）
- `5xx`：服務器錯誤（500, 502, 503, 504 等）

**輸出格式**：`[WARNING] 異常狀態 (HTTP {code})`

**含義**：
- DNS 解析正常
- 網絡連線正常
- 但網站應用層有問題

**常見情況**：
- `500`：服務器內部錯誤（程式 bug、配置錯誤）
- `502`：網關錯誤（後端服務掛了）
- `503`：服務不可用（維護中或過載）
- `504`：網關超時（後端響應太慢）
- `404`：頁面不存在（可能是配置錯誤）

**建議動作**：
- 通知技術團隊檢查服務器
- 不是封鎖問題，是網站本身的問題

---

### ❌ API_ERROR - 檢測工具失敗
**定義**：檢測工具本身出錯，無法完成檢測
**判斷條件**：
- API 請求失敗（無法創建測量任務）
- API 返回錯誤（429 Too Many Requests）
- 網絡連線問題
- JSON 解析失敗

**輸出格式**：`[API_ERROR] ❌ 檢測失敗 - {原因}`

**常見原因**：
1. **頻率限制**：`API 建立失敗，可能觸發頻率限制`
2. **配額用盡**：API 免費額度用完
3. **網絡問題**：本地網絡連線問題
4. **API 服務故障**：第三方 API 服務掛了

**含義**：
- **不代表域名有問題**
- 只是檢測工具無法完成檢測
- 需要重試或調整檢測策略

**建議動作**：
- 降低檢測頻率
- 增加延遲時間
- 實施重試機制
- 分批檢測

---

### 🔄 PARTIAL - 部分節點異常
**定義**：多個檢測節點結果不一致
**判斷條件**：
- 部分節點 CLEAN，部分節點 TIMEOUT
- 部分節點 CLEAN，部分節點 BLOCKED
- 不同節點解析到不同 IP

**輸出格式**：`[PARTIAL] 🔄 部分節點異常`

**含義**：
- 區域性封鎖（某些 ISP 封鎖，某些正常）
- DNS 解析不一致
- 網絡路由問題

**範例**：
- `dubaibet.club`：1 個節點 TIMEOUT，2 個節點 CLEAN
- `hod77.com`：2 個節點 CLEAN，1 個節點 TIMEOUT

**建議動作**：
- 記錄哪些 ISP/地區有問題
- 考慮使用多個 CDN 節點
- 監控特定地區的可用性

---

## 2. 優先級排序

從高到低：

1. **🔴 BLOCKED** - 最嚴重，需要立即處理
2. **🔄 PARTIAL** - 部分地區有問題，需要關注
3. **⚠️ TIMEOUT（全部節點）** - 域名完全無法訪問
4. **⚠️ WARNING** - 網站應用層問題
5. **🟢 CLEAN** - 正常
6. **❌ API_ERROR** - 工具問題，不是域名問題

## 4. 實施建議

### 4.1 檢測流程
```
1. 發起 API 請求
   ↓
2. 檢查 API 是否成功
   ├─ 失敗 → [API_ERROR]
   └─ 成功 → 繼續
   ↓
3. 等待結果返回
   ↓
4. 解析每個節點的結果
   ├─ DNS 污染 IP → [BLOCKED]
   ├─ 無 IP 或狀態碼 0 → [TIMEOUT]
   ├─ 2xx/3xx/403 → [CLEAN]
   └─ 其他狀態碼 → [WARNING]
   ↓
5. 綜合判斷
   ├─ 所有節點一致 → 使用該狀態
   └─ 節點不一致 → [PARTIAL]
```

### 4.2 重試策略
```
API_ERROR 情況：
- 第 1 次失敗：等待 2 秒，重試
- 第 2 次失敗：等待 5 秒，重試
- 第 3 次失敗：等待 10 秒，重試
- 仍失敗：標記為 API_ERROR，記錄到失敗清單

TIMEOUT 情況：
- 不重試（可能是域名本身問題）
- 記錄到報告中

BLOCKED 情況：
- 不重試（確認是封鎖）
- 記錄到高優先級清單
```

### 4.3 頻率控制
```
- 每個域名檢測後延遲：3-5 秒（可配置）
- 批次大小：50 個域名/批次
- 批次間延遲：30 秒
- 並發數：1（串行檢測，避免觸發限制）
- 每小時最大檢測數：600 個（根據 API 限制調整）
```

## 5. 配置文件範例

```bash
# 檢測配置
DELAY_BETWEEN_DOMAINS=3        # 每個域名之間的延遲（秒）
DELAY_AFTER_API_ERROR=10       # API 錯誤後的延遲（秒）
BATCH_SIZE=50                  # 批次大小
BATCH_DELAY=30                 # 批次間延遲（秒）
MAX_RETRIES=3                  # 最大重試次數
RETRY_BACKOFF_BASE=2           # 重試退避基數（秒）

# API 配置
API_WAIT_TIME=8                # 等待 API 結果的時間（秒）
API_TIMEOUT=15                 # API 請求超時時間（秒）

# 污染 IP 列表（可擴展）
BLOCKED_IPS=(
    "36.86.63.185"
    "10.0.0.0/8"
    "127.0.0.1"
    "0.0.0.0"
)

# 輸出配置
ENABLE_REALTIME_OUTPUT=true    # 是否顯示即時輸出
ENABLE_SUMMARY=true            # 是否顯示摘要
ENABLE_CSV_EXPORT=true         # 是否導出 CSV
CSV_OUTPUT_DIR="./reports"     # CSV 輸出目錄
```
