## 👥 應變組織架構

所有 BCP 劇本採用統一的應變組織架構：

| 角色 | 職責 | 聯絡方式 |
|------|------|----------|
| **事件指揮官** | 統籌決策、對外溝通 | Slack |
| **IDC 工程師** | 執行技術切換、驗證解析、監控系統狀態 | 啟用備援機制 / @sysadmin |
| **產品AM** | 協助備援 DNS 設定、評估影響範圍 | 備援：@network-team |
| **產品客服** | 客戶溝通、公告發布 | @support-lead |

## 🚨 標準應變流程

所有 BCP 劇本遵循以下標準流程：

### 階段一：確認與通報（0-5 分鐘）
- 確認異常範圍
- 啟動事件通報（Slack #infra-alert、通報流程告警、電話通知）
- 對外公告（狀態頁、TG）

### 階段二：緊急處理（5-30 分鐘）
- 執行技術修復措施
- 驗證修復結果

### 階段三：監控主要DNS 是否恢復
- 持續監控恢復狀態

### 階段四：通知產品AM 主要DNS已經恢復，安排切換時間
- 通知相關團隊
- 安排後續處理

### 階段五：BCP 結束通報
- 正式結束 BCP 流程
- 更新狀態頁

### 階段六：Retro
- 事後檢討會議
- 記錄改進項目

## 🔍 偵測機制

### 自動偵測標準
- **外部監控服務**：外部監控回報異常
- **內部監控**：連續 3 次 API or 其他域名呼叫失敗（HTTP 503/500/Timeout）
- **觸發門檻**：3 個不同來源同時回報異常，持續 ≥ 3 分鐘

### 手動偵測標準
- 執行 `dig +trace example.com` 追蹤解析路徑
- 檢查 Cloudflare/AWS Route53 錯誤訊息
- 檢查 DNS 記錄狀態

## 📢 通報機制

### 內部通報
- **Slack**：`#infra-alert` 頻道發布緊急通知
- **告警系統**：觸發通報流程告警（P0/P1 級別）
- **電話通知**：事件指揮官、產品AM、IDC

### 對外公告
- **狀態頁**：`請填寫`（需填入實際狀態頁 URL）
- **TG 公告**：發布 Telegram 公告

## 📝 使用指南

1. **發生異常時**：
   - 立即參考對應的 BCP 劇本
   - 按照階段流程執行應變措施
   - 記錄所有決策與時間點

2. **定期演練**：
   - 每季度至少演練一次 BCP 流程
   - 驗證通報機制與聯絡方式
   - 更新劇本內容

3. **持續改進**：
   - 每次事件後進行 Retro
   - 更新改進項目到對應劇本
   - 分享經驗給團隊
