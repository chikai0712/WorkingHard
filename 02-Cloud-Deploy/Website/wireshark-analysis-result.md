## ❓ 為什麼沒有看到故障切換？

### 可能原因

1. **DNS 沒有被封鎖**
   - pfctl 規則可能沒有生效
   - 或者測試時沒有封鎖

2. **DNS 快取**
   - 結果被快取，不查詢 DNS
   - 需要清除快取才能看到新查詢

3. **macOS 並行查詢機制**
   - macOS 可能同時查詢所有配置的 DNS
   - 不等待第一個失敗就查詢第二個

4. **測試時機**
   - 可能在封鎖解除後才捕獲封包
   - 或者封鎖規則沒有正確應用

## 🔧 如何看到真正的故障切換？

### 步驟 1：確認封鎖生效

```bash
# 封鎖 AWS DNS
sudo pfctl -a custom -f - <<< 'block drop out quick proto udp from any to 205.251.197.44 port 53'

# 驗證規則
sudo pfctl -sr | grep 205.251.197.44
# 應該看到封鎖規則

# 測試封鎖
dig @205.251.197.44 www.clouddeployment168.site +time=2
# 應該失敗或超時
```

### 步驟 2：清除快取並捕獲

```bash
# 清除 DNS 快取
sudo dscacheutil -flushcache
sudo killall -HUP mDNSResponder

# 開啟 Wireshark，過濾器：dns || icmp

# 執行查詢（不指定 DNS）
nslookup www.clouddeployment168.site
```

### 步驟 3：在 Wireshark 中應該看到

1. **DNS Query** → 205.251.197.44
2. **ICMP Destination Unreachable** ← 205.251.197.44（表示被封鎖）
3. **DNS Query Retransmission** → 205.251.197.44（可選，系統重試）
4. **DNS Query** → 216.239.32.108 或 8.8.8.8（切換到其他 DNS）
5. **DNS Response** ← 成功回應

**時間間隔：**
- 從第一次查詢到切換：應該有明顯延遲（1-5 秒）
- 如果立即切換，可能是並行查詢

## 📝 建議

### 1. 驗證封鎖是否生效

在測試前，確認：
- pfctl 規則存在
- 直接查詢被封鎖的 DNS 會失敗
- 清除 DNS 快取

### 2. 只配置一個 DNS（測試用）

如果系統配置了多個 DNS，可能使用並行查詢：
1. 系統設定 → 網路 → Wi-Fi → 詳細資訊 → DNS
2. 移除所有 DNS，只保留一個（例如：205.251.197.44）
3. 封鎖這個 DNS
4. 測試切換

### 3. 觀察時間間隔

真正的故障切換應該有明顯的時間延遲：
- 第一次查詢失敗：立即或幾百毫秒
- 系統等待/重試：1-3 秒
- 切換到第二個 DNS：1-5 秒後

如果看到立即切換（< 100ms），可能是並行查詢。
