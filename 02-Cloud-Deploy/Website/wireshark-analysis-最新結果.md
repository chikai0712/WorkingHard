## ❓ 為什麼沒有看到故障切換？

### 可能原因

1. **封鎖未生效**
   - pfctl 規則可能沒有正確應用
   - 需要驗證規則是否存在

2. **查詢在封鎖前執行**
   - 可能在封鎖前就已經查詢了 DNS
   - 需要先封鎖，再清除快取，再查詢

3. **macOS 並行查詢機制**
   - macOS 可能同時查詢所有配置的 DNS
   - 不等待第一個失敗就查詢第二個

## 🔧 如何看到真正的故障切換？

### 步驟 1：確認封鎖生效

```bash
# 1. 封鎖 DNS
echo 'block drop out quick proto udp from any to 205.251.197.44 port 53' | \
    sudo pfctl -a custom -f -

# 2. 驗證規則存在
sudo pfctl -a custom -sr | grep 205.251.197.44
# 必須看到規則！

# 3. 測試封鎖（必須失敗）
dig @205.251.197.44 www.clouddeployment168.site +time=2 +tries=1
# 應該失敗或超時
```

**重要：** 如果 `dig` 仍然成功，表示封鎖未生效！

### 步驟 2：清除快取並重新測試

```bash
# 1. 清除 DNS 快取
sudo dscacheutil -flushcache
sudo killall -HUP mDNSResponder

# 2. 等待 2 秒（確保快取清除）

# 3. 開啟 Wireshark，過濾器：dns || icmp

# 4. 執行查詢（不指定 DNS）
nslookup www.clouddeployment168.site
```

### 步驟 3：在 Wireshark 中應該看到

**正確的故障切換應該看到：**

1. **DNS Query** → 205.251.197.44
   - 時間：例如 0.000s

2. **ICMP Destination Unreachable** ← 205.251.197.44
   - 時間：例如 0.001s
   - **這是關鍵！** 表示 DNS 被封鎖

3. **DNS Query Retransmission** → 205.251.197.44（可選）
   - 時間：例如 1.000s
   - 系統重試

4. **DNS Query** → 8.8.8.8 或 216.239.32.108
   - 時間：例如 2.000s
   - **明顯的時間延遲（1-5 秒）**

5. **DNS Response** ← 成功回應
   - 時間：例如 2.050s

**時間間隔：**
- 第一次查詢到 ICMP 錯誤：< 10ms
- ICMP 錯誤到切換：**1-5 秒**（這是關鍵！）
- 切換到成功回應：< 100ms
